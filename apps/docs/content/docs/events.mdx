The system uses events at two levels: **EventBridge events** to trigger sequences and emails, and **SES engagement events** to track delivery, opens, clicks, bounces, complaints, and replies.

## EventBridge events

Your app publishes events to a custom EventBridge bus. Four types of routing rules handle them:

### Sequence triggers

An EventBridge rule matches a `detailType` and starts a Step Functions execution:

```json
{
  "source": ["your-app"],
  "detail-type": ["customer.created"]
}
```

The `subscriberMapping` in the sequence config extracts subscriber fields from the event using JSONPath:

```typescript
subscriberMapping: {
  email: "$.detail.email",
  firstName: "$.detail.firstName",
  attributes: "$.detail",
}
```

This means your event payload at `detail` becomes the subscriber's attributes. Any fields you include (platform, country, plan, etc.) are available for choice branching and Liquid template rendering.

### Fire-and-forget events

One-off emails triggered by events, with no sequence or state machine:

```typescript
events: [
  {
    detailType: "customer.first_sale",
    templateKey: "onboarding/first-sale-congrats",
    subject: "Congrats on your first sale!",
  },
];
```

These create EventBridge rules that invoke SendEmailFn directly. The Lambda upserts the subscriber profile and sends the email in one call.

### Subscribe events

Explicit subscriber registration with tags and attributes. Publishes to SubscribeFn:

```json
{
  "source": ["your-app"],
  "detail-type": ["subscriber.subscribed"]
}
```

The event detail must include `email` and `firstName`. Optionally include `attributes` (key-value pairs) and `tags` (string array):

```typescript
await eventBridgeClient.send(
  new PutEventsCommand({
    Entries: [
      {
        Source: "your-app",
        DetailType: "subscriber.subscribed",
        EventBusName: "mailshot-bus",
        Detail: JSON.stringify({
          email: "user@example.com",
          firstName: "Jane",
          attributes: { plan: "pro", country: "US" },
          tags: ["product-updates", "beta-users"],
        }),
      },
    ],
  }),
);
```

SubscribeFn upserts the subscriber profile and syncs the [tag inverted index](/docs/subscribers#tags). If the subscriber already exists, attributes and tags are updated.

### Broadcasts

One-off emails sent to a filtered group of subscribers. Think product updates, announcements, or newsletters. Triggered by invoking BroadcastFn directly (via AWS Lambda SDK or MCP tool):

```typescript
import { LambdaClient, InvokeCommand } from "@aws-sdk/client-lambda";

const lambda = new LambdaClient({});
const result = await lambda.send(
  new InvokeCommand({
    FunctionName: "Mailshot-broadcast",
    Payload: Buffer.from(
      JSON.stringify({
        broadcastId: "product-update-2026-04",
        templateKey: "broadcasts/product-update-april",
        subject: "What's new in April",
        sender: {
          fromEmail: "updates@yourdomain.com",
          fromName: "Your Product",
          replyToEmail: "support@yourdomain.com",
        },
        filters: {
          tags: ["product-updates"],
          attributes: { plan: "pro" },
        },
      }),
    ),
  }),
);
// result.Payload → { broadcastId, subscriberCount, dryRun }
```

#### How broadcasts work

```
Lambda.invoke(BroadcastFn)
  → BroadcastFn (resolves subscriber list, writes broadcast record)
    → SQS queue (one message per subscriber)
      → SendEmailFn (sends email, tracks engagement)
```

1. **BroadcastFn** receives the payload and resolves the subscriber list based on filters
2. Writes a broadcast record to DynamoDB (`PK = BROADCAST`, `SK = <timestamp>#<broadcastId>`)
3. For each matching subscriber, it publishes an SQS message with a `fire_and_forget` payload
4. **SQS** provides natural backpressure, retries (3 attempts), and a dead-letter queue for failed sends
5. **SendEmailFn** processes each message: runs pre-send checks, renders template, sends via SES, writes send log
6. Returns `{ broadcastId, subscriberCount, dryRun }` to the caller

Set `dryRun: true` in the payload to resolve the subscriber list and return the count without actually sending emails or writing a broadcast record. Useful for previewing audience size before committing.

The `broadcastId` flows through as the `sequenceId` in engagement tracking, so you can query opens/clicks/bounces per broadcast via the [MCP engagement tools](/docs/events#querying-engagement-via-mcp).

#### Filtering subscribers

Filters support **tags** and **attributes**, applied together as AND conditions.

| Filter       | How it works                                                                                                             | Example                             |
| ------------ | ------------------------------------------------------------------------------------------------------------------------ | ----------------------------------- |
| `tags`       | Queries the tag inverted index (`TAG#<name>` items). Multiple tags use AND logic — subscriber must have ALL listed tags. | `["product-updates", "enterprise"]` |
| `attributes` | Equality match on subscriber profile attributes.                                                                         | `{ plan: "pro", country: "US" }`    |

When both `tags` and `attributes` are specified:

1. Tag inverted index is queried first (efficient, narrows the set)
2. Matching subscriber profiles are batch-fetched
3. Attribute filters are applied in memory
4. Unsubscribed and suppressed subscribers are excluded automatically

When no filters are specified, all active subscribers receive the broadcast.

#### Broadcast sender config

Every broadcast must include a `sender` object:

```typescript
sender: {
  fromEmail: "updates@yourdomain.com",  // SES-verified address
  fromName: "Your Product",             // Display name
  replyToEmail: "support@yourdomain.com", // Optional Reply-To
  listUnsubscribe: true,                // Default: true
}
```

This follows the same [per-sequence sender config](/docs/sequences) pattern. Set `listUnsubscribe: false` to omit `List-Unsubscribe` headers (the in-body unsubscribe link is still available).

#### Broadcast templates

Templates for broadcasts follow the same format as sequence templates (React Email → HTML, rendered with LiquidJS). Store them under a `broadcasts/` prefix:

```
templates/broadcasts/product-update-april.html
```

All the same template variables are available: `email`, `firstName`, custom attributes, `unsubscribeUrl`, `currentYear`, and display names.

### Publishing events

From your app, publish to EventBridge:

```typescript
await eventBridgeClient.send(
  new PutEventsCommand({
    Entries: [
      {
        Source: "your-app",
        DetailType: "customer.created",
        EventBusName: "mailshot-bus",
        Detail: JSON.stringify({
          email: "user@example.com",
          firstName: "Jane",
          platform: "kajabi",
          country: "ZA",

      },
    ],
  }),
);
```

## SES engagement tracking

SES publishes engagement events to SNS, which triggers EngagementHandlerFn. Six event types are tracked:

| Event type  | What it means                                                       |
| ----------- | ------------------------------------------------------------------- |
| `delivery`  | SES successfully delivered the email to the recipient's mail server |
| `open`      | Recipient opened the email (tracking pixel)                         |
| `click`     | Recipient clicked a tracked link                                    |
| `bounce`    | Email bounced (permanent or transient)                              |
| `complaint` | Recipient marked the email as spam                                  |
| `reply`     | Recipient replied to the email (requires inbound reply handling)    |

### Events table

Engagement events are stored in a dedicated DynamoDB table:

```
PK: SUB#user@example.com
SK: EVT#2026-03-17T10:30:00.000Z#delivery
```

Each event record includes:

- `eventType` - delivery, open, click, bounce, complaint, reply
- `templateKey` - Which template was sent
- `sequenceId` - Which sequence the email belonged to
- `subject` - Email subject line
- `timestamp` - When the event occurred
- `ttl` - Optional, set via `DATA_TTL_DAYS` env var (unset = permanent)

Reply events additionally include:

- `body` - Plain text content of the reply
- `sesMessageId` - The original outbound message ID (from `In-Reply-To` header, best-effort)

### TemplateIndex GSI

A global secondary index enables querying events by template:

```
GSI PK: templateKey (e.g., "onboarding/welcome")
GSI SK: EVT#<timestamp>#<eventType>
```

This powers cross-subscriber analytics: "How many people opened the welcome email this week?"

### Bounce and complaint handling

Bounces and complaints flow through two paths:

1. **EngagementHandlerFn** - Records the event in the Events table for analytics
2. **BounceHandlerFn** - Suppresses the subscriber and stops all executions

BounceHandlerFn only acts on permanent bounces and complaints. Transient bounces are recorded but don't trigger suppression.

## Inbound reply tracking

When `REPLY_TO_EMAIL` is set in `.env` and the receiving domain has MX records pointing to SES, inbound replies are automatically tracked as engagement events.

### How it works

```
Inbound Reply → SES Receipt Rule → SNS → ReplyHandlerFn
  1. Parses raw email from SNS notification
  2. Matches sender to subscriber in DynamoDB
  3. Writes reply event to Events table (eventType: "reply")
  4. Publishes "email.replied" to EventBridge
```

ReplyHandlerFn performs best-effort correlation: if the reply's `In-Reply-To` header contains the original SES message ID, it queries the send log to tag the reply with the `sequenceId` and `templateKey` of the email being replied to. This isn't guaranteed (not all mail clients include `In-Reply-To`).

### Stopping sequences on reply

Any sequence can opt into stopping when a subscriber replies by using `exitOn`:

```typescript
exitOn: [
  {
    detailType: "email.replied",
    subscriberMapping: {
      email: "$.detail.email",
      firstName: "$.detail.email",
    },
  },
],
```

This uses the existing `exitOn` mechanism - an EventBridge rule routes `email.replied` events to SequenceExitFn, which stops the execution. See [Sequences - Exit events](/docs/sequences) for details.

### Forwarding replies to your inbox

By default, captured replies are processed (stored as events, published to EventBridge) but not forwarded anywhere. To receive replies in your real inbox so you can continue the conversation, add `forwardRepliesTo` to your sequence's sender config:

```typescript
sender: {
  fromEmail: "you@yourdomain.com",
  fromName: "Your Name",
  replyToEmail: "replies@yourdomain.com",
  captureReplies: true,
  forwardRepliesTo: "your-real-inbox@gmail.com",
},
```

When a reply arrives, ReplyHandlerFn:

1. Processes it normally (stores event, publishes to EventBridge)
2. Forwards the raw email to `forwardRepliesTo` via SES
3. Rewrites the `From:` header to the verified `replyToEmail` address (SES requires a verified sender)
4. Sets `Reply-To:` to the original sender's address, so you can reply directly back to them

Forwarding is fire-and-forget — if it fails, the reply event is still recorded and the sequence still stops (if `exitOn` is configured). Errors are logged but don't affect the rest of the handler.

### DNS requirements

SES inbound email requires MX records pointing to SES for the receiving domain. Add an MX record for your `REPLY_TO_EMAIL` domain:

```
MX 10 inbound-smtp.<region>.amazonaws.com
```

Replace `<region>` with your AWS region (e.g., `us-east-1`). SES receipt rules are only available in certain regions (us-east-1, us-west-2, eu-west-1, and a few others).

## Querying engagement via MCP

The MCP server provides engagement query tools:

### get_subscriber_events

Query events for a specific subscriber:

```
get_subscriber_events(email: "user@example.com", eventType: "open", startDate: "2026-03-01")
```

### get_template_events

Query events across all subscribers for a template (uses TemplateIndex GSI):

```
get_template_events(templateKey: "onboarding/welcome", eventType: "click")
```

### get_sequence_events

Query events for an entire sequence or broadcast (uses SequenceIndex GSI — efficient, no scan):

```
get_sequence_events(sequenceId: "onboarding", startDate: "2026-03-01", endDate: "2026-03-17")
```

For broadcasts, use the `broadcastId` as the `sequenceId`:

```
get_sequence_events(sequenceId: "product-update-2026-04")
```

### get_delivery_stats

Aggregate delivery statistics over a date range:

```
get_delivery_stats(startDate: "2026-03-01", endDate: "2026-03-17")
```

Returns counts by event type: deliveries, opens, clicks, bounces, complaints.

## Broadcast management via MCP

### send_broadcast

Invoke BroadcastFn directly. Returns subscriber count and messages queued:

```
send_broadcast(
  broadcastId: "product-update-2026-04",
  templateKey: "broadcasts/product-update-april",
  subject: "What's new in April",
  senderFromEmail: "updates@yourdomain.com",
  senderFromName: "Your Product",
  filterTags: ["product-updates"]
)
```

### get_broadcast

Retrieve a broadcast record showing config, filters, subscriber count, and send time:

```
get_broadcast(broadcastId: "product-update-2026-04")
```

### list_broadcasts

List recent broadcasts, most recent first:

```
list_broadcasts(limit: 10)
```

### Broadcast engagement tracking

Broadcasts use the existing engagement tracking system. The `broadcastId` is stored as the `sequenceId` on all engagement events, so you can query broadcast stats the same way you query sequence stats:

| Metric      | How to query                                                              |
| ----------- | ------------------------------------------------------------------------- |
| Deliveries  | `get_sequence_events(sequenceId: "<broadcastId>", eventType: "delivery")` |
| Opens       | `get_sequence_events(sequenceId: "<broadcastId>", eventType: "open")`     |
| Clicks      | `get_sequence_events(sequenceId: "<broadcastId>", eventType: "click")`    |
| Bounces     | `get_sequence_events(sequenceId: "<broadcastId>", eventType: "bounce")`   |
| By template | `get_template_events(templateKey: "broadcasts/<name>")`                   |

All queries use GSIs (SequenceIndex or TemplateIndex) for efficient lookups — no table scans.
